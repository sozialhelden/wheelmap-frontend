# App State Management

## Overview

App state is **URL-driven global state**. Most pieces of state map to a query parameter so that reloading the page or sharing a link reproduces the exact same view. Some values are additionally persisted to `localStorage` so they survive across sessions even when they are not present in the URL.
Some values are persist-only and intentionally never appear in the URL (e.g. `onboardingCompleted`) — these are stored in `localStorage` and merged into the app state on every page load, but their serializers return `undefined` so they are excluded from the URL.“

The system is made up of four layers:

| Layer | File | Responsibility |
|---|---|---|
| Schema & config | `src/modules/app-state/app-state.tsx` | Defines every state key, its parser, serializer, default and persistence flag |
| Serialization | `src/modules/app-state/utils/query.ts` | Converts between URL search params and the `AppState` object |
| Persistence | `src/modules/app-state/utils/persistence.ts` | Reads/writes `localStorage` |
| React integration | `src/modules/app-state/hooks/useAppState.tsx` | Provides the `AppStateContextProvider` and the `useAppState()` hook |

A small helper in `src/modules/app-state/utils/app-state.ts` ties the layers together by merging defaults, persisted values and URL params into a single `AppState` object.

## State keys

The `appStateConfig` object in `app-state.tsx` defines every key:

| Key | Type | Default | Persisted? | In URL? |
|---|---|---|---|---|
| `needs` | `NeedSelection` | *(none)* | ✅ | ✅ |
| `position` | `MapPosition` | `{ latitude: 52.5, longitude: 13.3, zoom: 10 }` | ✅ | ✅ |
| `search` | `string` | *(none)* | ❌ | ✅ |
| `category` | `Category \| ""` | *(none)* | ❌ | ✅ |
| `onboardingCompleted` | `boolean` | `false` | ✅ | ❌ |
| `shouldLocateUser` | `boolean` | `false` | ✅ | ❌ |

### `needs`

The user's selected accessibility need filters. Contains a `NeedSelection` object — a record mapping need category keys (e.g. `mobility`, `toilet`) to a specific need level (e.g. `"fully-wheelchair-accessible"`, `"not-wheelchair-accessible"`, `"no-data"`). The available categories and levels are defined in `src/modules/needs/needs.ts`. When set, the map and place listings filter results to only show places matching the selected accessibility levels. Persisted to `localStorage` so the user's filter preferences survive across sessions. Also supports a legacy format that migrates from the old `wheelchair` and `toilet` query parameters used in earlier versions of the app.

### `position`

The current map viewport. Contains a `MapPosition` object with `latitude`, `longitude` and `zoom` — all optional numbers. Defaults to Berlin (`52.5, 13.3, zoom 10`) when neither the URL nor `localStorage` provides a value. Serialized with dynamic precision that increases with zoom level (via `lerp`) to keep URLs short at low zooms and precise at high zooms. Persisted to `localStorage` so the map returns to the user's last-viewed location. Also supports legacy `lat`, `lon` and `zoom`/`z` query parameter formats.

### `search`

The current text search query entered by the user in the search bar. A plain trimmed string (e.g. `"Berlin Hauptbahnhof"`). Not persisted — a fresh page load starts with no active search. Also supports the legacy `q` query parameter format.

### `category`

The active place category filter (e.g. `"restaurant"`, `"cafe"`, `"hospital"`). A string that must be a valid key from the category registry provided by `@sozialhelden/core` via `getCategories()`. When set, only places belonging to that category are shown on the map. Not persisted — navigating away or reloading without the query parameter clears the filter.

### `onboardingCompleted`

Whether the user has completed the onboarding flow. A boolean (`true`/`false`). Defaults to `false`. Persisted to `localStorage` so returning users are not shown the onboarding dialog again. **Never serialized to the URL** (the serializer returns `undefined`) — this is a device-local preference. Sharing a link will not transfer onboarding status; the recipient sees the onboarding on their own device if they haven't completed it yet.

### `shouldLocateUser`

Whether the app should automatically trigger the browser's geolocation API to locate the user on the map. A boolean (`true`/`false`). Defaults to `false`. Set to `true` at the end of the onboarding flow when the user shares their location. The `MapComponent` watches this value and calls `geoControlRef.current?.trigger()` when it becomes `true` (except on Mobile Safari, where geolocation permissions are session-scoped and triggering would re-show the native prompt on every reload). Persisted to `localStorage` so the app remembers the user's preference across sessions. **Never serialized to the URL** — sharing a link should not trigger geolocation on someone else's device.

---

### Config properties

Each key in `appStateConfig` can define the following properties:

- **`parser`** – Validates and transforms the raw query parameter value into a typed value.
- **`serializer`** *(optional)* – Converts the typed value back to a query-string-compatible value. If the serializer returns `undefined` the key is excluded from the URL entirely (used for persist-only values like `onboardingCompleted` and `shouldLocateUser`).
- **`legacy`** *(optional)* – Parses old/legacy query parameter formats for backwards compatibility.
- **`defaultValue`** *(optional)* – Fallback value when neither the URL nor `localStorage` provides a value.
- **`persist`** *(optional)* – Whether the value should be saved to `localStorage`.

## How state is assembled

`getAppStateFromSearchParams()` in `utils/app-state.ts` merges three sources in ascending priority:

```
defaults  ←  localStorage  ←  URL query params
```

In code:

```ts
{
  ...getDefaultAppState(),          // from config defaultValue entries
  ...getAppStateFromPersistence(),  // from localStorage
  ...parseQuery(query),             // from URL search params
}
```

URL query parameters always win. Persisted values win over defaults. This means a user who shares a link will see the linked state, while their own persist-only preferences (like `onboardingCompleted`) are still restored from their own browser.

---

## Query parameter format

Nested values use bracket notation in the URL. For example the `position` key produces:

```
?position[latitude]=52.5&position[longitude]=13.3&position[zoom]=10
```

`flattenToSearchParams()` and `unflattenSearchParams()` in `src/utils/search-params.ts` handle the conversion between nested objects and flat key-value pairs.

---

## Retrieving state (reading)

Components access state through the React context hook:

```tsx
const { appState } = useAppState();
// appState.position.latitude, appState.needs, appState.shouldLocateUser, etc.
```

`appState` is a `useMemo` value that recomputes whenever the Next.js `searchParams` change **or** whenever a revision counter is bumped (see *Persist-only updates* below).

---

## Altering state (writing)

```tsx
const { setAppState } = useAppState();

setAppState(
  { position: { latitude: 48.8, longitude: 2.3, zoom: 12 } },
  { routerOperation: "replace" },
);
```

`setAppState` accepts a **partial** `AppState` — only the keys you want to change — and an optional options object:

| Option | Default | Effect |
|---|---|---|
| `routerOperation: "push"` | ✅ | Adds a new browser history entry |
| `routerOperation: "replace"` | | Replaces the current history entry |
| `routerOperation: "shallow"` | | Updates the URL via `window.history.replaceState` without triggering a Next.js navigation |
| `keepExistingQuery: true` | ✅ | Merges new params into the existing query string |
| `keepExistingQuery: false` | | Builds the URL from scratch (only the new state params) |

### What `setAppState` does internally

1. Merges the new partial state into the current `appState`.
2. Calls `persistAppState()` immediately — this writes all persist-flagged keys to `localStorage`.
3. Serializes the merged state into query parameters and builds a new URL.
4. If the URL changed, navigates via `router.push`, `router.replace` or `window.history.replaceState`.
5. If the URL did **not** change (persist-only update), bumps a revision counter so the `useMemo` recomputes `appState` from `localStorage` and triggers a re-render.

---

## Persist-only updates

Some keys (`onboardingCompleted`, `shouldLocateUser`) intentionally never appear in the URL — their serializers return `undefined`. When `setAppState` changes only these keys the resulting URL is identical to the current one.

To ensure React still sees the change, `setAppState` increments a `revision` counter. This counter is included in the `useMemo` dependency array that computes `appState`, forcing it to re-read from `localStorage` and return the updated values.

Without the revision counter, changes to persist-only values would be written to `localStorage` but the React tree would never re-render, because `searchParams` (the primary dependency) didn't change.

---

## When state is updated automatically

The `AppStateContextProvider` contains three `useEffect` hooks that run automatically:

1. **Legacy migration** *(every render)* — checks if the current URL contains legacy query parameter formats (via each key's `legacy` function). If found, rewrites them into the modern format with `setAppState({ ... }, { keepExistingQuery: false })`.

2. **On mount** — calls `setAppState(appState, { keepExistingQuery: false, routerOperation: "shallow" })` once. This populates the URL with default and persisted values that were missing from the initial request (e.g. a first-time visitor gets `?position[latitude]=52.5&…` added to the URL).

3. **On every `appState` change** — calls `persistAppState(appState)` to keep `localStorage` in sync with the latest state.

## Data flow diagram

```
URL query string
       │
       ▼
 unflattenSearchParams()  →  parseQuery()
       │                          │
       │                          ▼
       │                  merge with defaults + localStorage
       │                          │
       ▼                          ▼
  React context (appState)  ◄───  getAppStateFromSearchParams()
       │
       │  setAppState({ ... })
       ▼
  getQueryFromAppState()  →  flattenToSearchParams()  →  new URL
       │                                                    │
       ├──► localStorage (persistAppState)                  │
       └──► router.push / replace / history.replaceState ◄──┘
              (or revision bump if URL unchanged)
```
