### Naming pattern

The project has these main folders:

- `docs/`: Documentation
- `public/`: Static files the web server serves directly
- `tests/`: Tests for user flows spanning multiple pages or the whole app
- `src/`: Main source code

Inside these folders, you'll see a common pattern:

- `components/`: Reusable UI components
- `utils/`: Utility functions
- `hooks/`: Custom hooks that use state or other hooks
- `types/`: Data types and interfaces
- `tests/`: Tests for the code in that folder

### Pages

Put components, hooks, types, utilities, or tests next to the page that uses
them. For example:

For `./src/app/upload/index.ts`, put related code in subfolders (`_components`,
`_hooks`, `_types`, `_tests`, `_utils`) inside `app/upload`.

### Modules

Put logic that spans multiple pages but belongs to a specific feature in
`./src/modules/`. For example:

We use categories to describe and filter places on different pages, so they live
in `./src/modules/categories`.

### Shared

Put code that multiple modules or pages share—but doesn't belong to a specific
feature—at the root of `src` in `components/`, `utils/`, `hooks/`, or `types/`.
Examples:

- UI components like `Button`
- Generic utilities for strings or dates
- Generic hooks for global state

## 2. Architecture

### 1. Global app state

The app has a global state that is entirely controlled by query parameters. E.g.
the current selection of the need picker, the selected category to filter places
by or the map bounds. The idea is to allow people to share links to the app to a
certain state.

You can configure the app-state in `src/modules/app-state/app-state.ts`. The
implementation also allows complex deeply nested objects as state as well as
persisting parts of it to local storage and restoring it when no query
parameters are found. As every state is different, you need to bring your own
parser that validates and sanitizes the contents, in case you want to add state
to the global one.
